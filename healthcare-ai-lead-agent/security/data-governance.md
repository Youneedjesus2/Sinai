# Data Governance Policy
Healthcare AI Lead Intake Agent

---

# 1. Purpose

This document defines how data is collected, stored, processed, and protected within the system.

The goal is to:

- minimize sensitive data storage
- protect user and agency data
- maintain clear data ownership
- ensure safe interaction with external services

The system is designed primarily for **lead intake and scheduling**, not for storing full patient medical records.

---

# 2. Data Categories

The system processes several categories of data.

## 2.1 Lead Contact Data

Information used to communicate with the lead.

Examples:

- name
- phone number
- email address
- preferred contact method

This data is necessary for communication and scheduling.

---

## 2.2 Conversation Data

Messages exchanged between the lead and the system.

Examples:

- inbound messages
- outbound responses
- timestamps
- channel type (SMS, email, form)

Conversation history is stored to maintain context and allow staff review.

---

## 2.3 Intake Information

Information describing the lead’s needs.

Examples:

- requested service
- location of care
- general description of care needs
- scheduling preferences

The system should only collect **high-level intake information** necessary to schedule a consultation.

---

## 2.4 Scheduling Data

Information used to manage consultations.

Examples:

- scheduled appointment time
- appointment status
- calendar provider ID
- rescheduling history

---

## 2.5 System Logs

Operational data generated by the system.

Examples:

- inbound message received
- model output validation result
- retrieval success or failure
- scheduling success or failure
- API errors

Logs are used for debugging and monitoring.

---

# 3. Data We Avoid Collecting

The system should avoid collecting or storing unnecessary sensitive data.

Examples of data to avoid storing:

- detailed medical history
- diagnoses
- medication lists
- insurance numbers
- social security numbers
- full patient records

If such information appears in messages, the system should store it only as part of conversation history and avoid copying it into structured fields.

---

# 4. Data Sources

Data enters the system through:

## External Users

Leads contacting the agency via:

- website forms
- SMS messages
- email messages

---

## Agency Administrators

Agency staff may provide:

- approved service descriptions
- FAQ documents
- intake policies
- scheduling rules

These documents become part of the RAG knowledge base.

---

## External Systems

External services may provide:

- calendar availability
- message delivery confirmations
- webhook events

---

# 5. Data Storage

The system stores data primarily in the application database.

## Database Storage

Stored records include:

- lead records
- conversation messages
- appointment records
- summaries
- audit logs

The database should enforce:

- authentication
- encrypted connections
- restricted access

---

## Vector Storage

Embeddings generated for RAG retrieval are stored in the vector index (pgvector).

Embeddings represent document meaning and do not directly expose full document content.

---

# 6. Data Access

Access to system data must be restricted based on role.

## Staff Access

Agency staff may access:

- leads assigned to their agency
- conversation history
- summaries
- appointment information

---

## System Services

Backend services may access:

- conversation data
- knowledge base documents
- scheduling records

Services should access data only when required for processing.

---

## External Vendors

External vendors may process limited data.

Examples:

### OpenAI

Processes conversation text and retrieved context for response generation.

---

### Twilio

Processes phone numbers and SMS content for message delivery.

---

### Email Provider

Processes email addresses and message content.

---

### Calendar Provider

Processes appointment details such as time and attendee information.

---

# 7. Data Retention

The system should avoid storing data indefinitely.

Recommended retention policy:

Conversation history:  
retain for **30–90 days**

Lead records:  
retain until consultation occurs or lead is closed

Logs:  
retain **30 days**

Summaries:  
retain as long as needed for consultation workflow

Retention policies may be adjusted based on agency needs.

---

# 8. Data Minimization

The system should collect only the data necessary to complete its purpose.

Examples:

Good practice:
- collect location city instead of full address
- collect general care need instead of medical diagnosis

Bad practice:
- collecting unnecessary personal or medical details

---

# 9. Logging Guidelines

Logs are important for debugging but should avoid exposing unnecessary sensitive information.

Logging should include:

- event type
- timestamp
- system action
- error details

Logging should avoid:

- full message bodies when unnecessary
- sensitive personal information

Where possible, logs should reference message IDs rather than raw content.

---

# 10. Data Protection

The system should implement basic protections:

- encrypted database connections
- secure storage of API keys
- restricted admin access
- authentication for dashboard access

Sensitive credentials must never be committed to source code.

---

# 11. Vendor Data Flow

When the system sends data to external vendors:

- only necessary fields should be transmitted
- API keys must be secured
- vendor usage should be monitored

Example flows:

User message → backend → OpenAI  
User phone number → backend → Twilio  
Appointment request → backend → calendar provider

---

# 12. Data Ownership

Lead data belongs to the healthcare agency using the system.

The platform should not reuse lead data for:

- training models
- analytics across agencies
- marketing

unless explicitly approved by the agency.

---

# 13. Data Deletion

The system should support deletion of lead records if requested by the agency.

Deletion should remove:

- lead records
- conversation history
- summaries
- scheduling data where applicable

Logs may retain anonymized system events.

---

# 14. Governance Review

This policy should be reviewed when:

- new data sources are introduced
- new integrations are added
- retention policies change
- system architecture changes
